# WHAT（要做什么）

OPC 一键任务调度系统 (`/opc`) 是 OPC Founder 的核心入口，提供智能任务评估和自动化 Agent 编排能力。用户只需输入一个任务描述，系统即可自动：

1. **评估任务类型** - 分析任务复杂度、涉及领域、依赖关系
2. **匹配工作流规范** - 根据关键词和模式自动匹配预定义工作流
3. **选择合适 Agent** - 从 29 个专业 Agent 中选择最佳组合
4. **编排执行模式** - 决定单 Agent、串行、并行或 Team 模式
5. **管理状态流转** - 跨会话持久化任务进度和知识积累

# WHY（为什么要做）

一人公司最大的挑战是**时间稀缺**。传统开发流程需要：
- 手动协调多个角色（产品、设计、开发、测试、运维）
- 在不同工具间切换（需求文档、设计工具、IDE、测试平台）
- 记忆和传递上下文（决策、约束、进度）

OPC 一键调度系统解决这些痛点：

| 痛点 | OPC 解决方案 |
|------|-------------|
| 手动协调耗时 | 自动编排，一键启动全流程 |
| 上下文丢失 | 知识库自动积累，跨阶段传递 |
| 工具切换成本 | 统一入口，Agent 自动调用工具 |
| 进度追踪困难 | 状态管理自动记录，随时查看 |

## 功能性需求

### 核心功能

1. **任务评估引擎**
   - 分析任务描述，识别领域（产品/设计/开发/测试/运维/增长）
   - 判断复杂度（简单/中等/复杂）
   - 检测依赖关系（串行/并行/混合）

2. **工作流匹配系统**
   - 关键词匹配：`实现`、`开发`、`修复`、`上线`等触发对应工作流
   - 正则模式匹配：支持复杂任务描述识别
   - 多匹配处理：多个候选时询问用户确认

3. **Agent 选择与编排**
   - 单 Agent 模式：简单任务直接调度
   - Pipeline 模式：串行执行，输出传递
   - Parallel 模式：独立任务并行执行
   - Team 模式：复杂项目，TeamCreate + TaskCreate

4. **状态管理集成**
   - 初始化：`opc_state_init` 创建任务会话
   - 进度追踪：`opc_state_read` 查看当前进度
   - 阶段更新：`opc_state_write` 记录完成状态
   - Agent 交接：`opc_handoff` 保留上下文

5. **知识库集成**
   - 自动初始化：`opc_knowledge_init` 创建知识域
   - 前置读取：Agent 调度前读取相关知识
   - 后置写入：Agent 完成后更新知识库
   - 特征匹配：相似任务自动复用知识

6. **命令支持**
   - `/opc <任务>` - 创建并执行任务
   - `/opc status` - 显示当前任务进度
   - `/opc resume` - 恢复上次活跃会话

### 辅助功能

1. **HUD 状态栏显示**
   - 当前模型、会话时长、活跃 Skill
   - 上下文使用率、工具调用统计

2. **进度可视化**
   - 阶段图标：✅ 完成、🔄 执行中、⏳ 待执行、🚫 阻塞
   - 产物展示：显示已生成的文件和文档
   - Agent 状态：当前工作的 Agent

## 非功能性需求

- **性能**：
  - 任务评估响应时间 < 2秒
  - Agent 调度延迟 < 1秒
  - 状态读写操作 < 100ms

- **安全性**：
  - 状态文件存储在用户本地 `.opc/state/` 目录
  - 不上传任何敏感信息到远程服务器
  - Git 提交时自动排除个人会话数据

- **可靠性**：
  - 单窗口单任务模型，避免状态混乱
  - 检查点机制支持回滚
  - 异常时保留当前状态，支持恢复

- **可用性**：
  - 一键入口，无需学习复杂命令
  - 自然语言任务描述，无需特定格式
  - 进度可视化，随时了解任务状态

## 不做什么（Non-goals）

1. **不支持多任务并行** - 一人公司专注单一任务更高效
2. **不替代专业工具** - Agent 使用现有工具，不重新发明
3. **不强制特定流程** - 工作流可自定义，适应不同团队
4. **不处理实时协作** - 单人工作，无多人协作需求
