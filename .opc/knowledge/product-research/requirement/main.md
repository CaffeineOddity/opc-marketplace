# WHAT（要做什么）

产品需求调研系统（product-research）是一套完整的产品阶段工具集，为一人公司提供从市场调研到需求定义再到规范驱动开发的完整能力。

核心功能包括：
- **市场与用户调研**：通过 `/research` skill 执行结构化调研，获取市场洞察
- **需求文档撰写**：通过 `/requirement` skill 生成结构化 PRD（产品需求文档）
- **结构化头脑风暴**：通过 `/brainstorm` skill 使用 SCAMPER/第一性原理/逆向思维等方法进行创意生成
- **规范驱动开发**：通过 `/spec-driven-development` skill 在实现前定义清晰的接口契约、数据模型和行为规则

# WHY（为什么要做）

一人公司模式下，产品阶段面临独特挑战：
- **资源有限**：无法组建完整的产品团队，需要一人承担多角色职责
- **效率优先**：需要快速迭代，不能陷入过度文档化的陷阱
- **决策聚焦**：需要从海量信息中快速提炼关键洞察
- **质量保障**：需求必须清晰、可测试，避免开发阶段的歧义

product-research 系统解决这些痛点：
- 提供结构化方法论，降低调研和需求撰写的门槛
- 自动化 PRD 生成，减少重复性文档工作
- 集成规范驱动开发，确保需求可测试、可追溯
- 支持与下游设计、开发阶段的无缝交接

## 功能性需求

### 1. 市场调研能力（/research skill）

**调研框架**：
- 定义调研问题：需要学习什么、目标用户是谁、需要验证的假设
- 市场格局分析：现有解决方案、竞争对手、市场规模与趋势、机会与空白
- 用户洞察：目标用户画像、痛点与未满足需求、用户行为模式
- 综合输出：关键发现摘要、已验证/未验证假设、下一步建议

**输出格式**：
- 结构化调研简报，包含清晰章节划分
- 优先提供可行动的洞察而非原始数据
- 明确标记不确定性

### 2. 需求撰写能力（/requirement skill）

**需求模板**：
- **概述**：问题陈述、目标用户、成功指标
- **用户故事**：As a [persona], I want [goal], so that [benefit] 格式
- **优先级**：Must-have / Should-have / Nice-to-have 分级
- **验收标准**：每个故事的可测试验收条件
- **功能需求**：核心功能详细行为、边缘情况与错误处理、数据模型与关系
- **非功能需求**：性能目标、安全考量、兼容性约束
- **范围与优先级**：MVP 范围（仅 Must-have）、后续阶段、明确排除项

**质量标准**：
- 开发者可直接实现，无歧义
- 每个需求可测试
- 描述 WHAT 而非 HOW

### 3. 头脑风暴能力（/brainstorm skill）

**会议结构**：
- **定义挑战**：清晰陈述问题或机会、定义约束与边界、设定明确目标
- **发散阶段**：无评判生成 10+ 创意，使用技术：
  - SCAMPER：替代、组合、调整、修改、其他用途、消除、逆向
  - 第一性原理：拆解到基础，重新构建
  - 类比法：其他行业/领域如何解决此问题
  - 逆向思维：什么会导致失败？反转它
- **收敛阶段**：按 Impact x Feasibility x Alignment 评分、归类相似创意、选出 Top 3
- **下一步**：每个 Top 创意的验证行动、定义"完成"标准、建议后续负责阶段/agent

**指导原则**：
- 发散阶段重数量而非质量
- 构建创意而非否定
- 接受大胆创意——常带来突破性洞察

### 4. 规范驱动开发（/spec-driven-development skill）

**核心原则**：无实现先定义规范。规范是契约。

**规范结构**：
- **接口契约**：API 端点、请求/响应格式、错误码
- **数据模型**：实体、属性、约束、索引
- **行为规则**：业务逻辑、状态转换、Given-When-Then 格式
- **边界条件**：输入验证、速率限制、分页行为

**SDD 流程**：
```
规范定义 → 规范评审 → 测试推导 → 实现 → 验证
```

**与 TDD 集成**：
- 每个规范项推导至少一个测试用例
- RED: 从规范编写失败测试
- GREEN: 最小实现通过测试
- REFACTOR: 保持绿色前提下清理
- 验证实现是否满足规范

### 5. Agent 能力

**product-agent**（sonnet 模型）：
- 核心职责：调研、需求撰写、创意生成
- 交接协议：
  - 输出给 design-agent：需求文档 + 用户故事 + 验收标准
  - 输出给 dev-agents：技术需求 + 数据模型 + API 规范
  - 输出给 marketing-agent：产品定位 + 价值主张
  - 输入来自 founder-agent：战略方向与优先级
  - 输入来自 qa-agent：用户反馈与 bug 报告
  - 输入来自 marketing-agent：市场信号与竞争情报

**startup-analyst**（inherit 模型）：
- 核心职责：市场规模分析（TAM/SAM/SOM）、财务建模、竞争分析、团队规划
- 适用场景：早期创业公司（pre-seed 到 Series A）
- 分析能力：
  - 市场规模：自下而上/自上而下/价值理论方法论
  - 财务建模：基于 cohort 的收入预测、单位经济分析（CAC/LTV/回本期）
  - 竞争分析：波特五力、蓝海战略、竞争定位
  - 团队规划：按阶段招聘计划、薪酬基准、股权分配

**business-analyst**（sonnet 模型）：
- 核心职责：业务流程分析、需求获取、干系人管理
- 分析技术：SWOT、根因分析、成本效益分析、风险评估、流程映射
- 文档能力：BRD、功能规格、流程图、用例图、数据流图

## 非功能性需求

- **性能**：
  - 调研响应时间：< 30秒生成结构化调研简报
  - 需求文档生成：< 60秒完成完整 PRD
  - 头脑风暴会议：< 2分钟完成发散-收敛流程

- **安全性**：
  - 调研数据不存储敏感用户信息
  - 需求文档版本控制，防止意外丢失
  - 与下游系统交接时保持数据完整性

- **可靠性**：
  - 调研结果可追溯至数据源
  - 需求文档可测试性 100%
  - 规范文档无歧义，单一解释

- **可用性**：
  - Skill 命令简洁易记：`/research`、`/requirement`、`/brainstorm`、`/spec-driven-development`
  - 输出格式结构化，易于阅读和编辑
  - 支持中英文双语输出

## 不做什么（Non-goals）

- **不做项目管理**：不替代项目管理工具（如 Jira），专注于需求阶段
- **不做设计**：不生成 UI 设计稿，交接给 design-kit
- **不做代码实现**：不生成实现代码，交接给 dev-kit
- **不做过度文档**：一人公司需要交付而非文档，保持需求精简
- **不做用户测试执行**：提供方法论但不执行实际用户测试

## 验收标准（Done Definition）

- [ ] `/research` skill 可执行完整调研流程并输出结构化简报
- [ ] `/requirement` skill 可生成包含用户故事和验收标准的完整 PRD
- [ ] `/brainstorm` skill 可使用至少 3 种方法进行结构化创意生成
- [ ] `/spec-driven-development` skill 可输出接口契约、数据模型、行为规则
- [ ] product-agent 可接收 founder-agent 指令并输出可交接的需求文档
- [ ] startup-analyst 可执行 TAM/SAM/SOM 分析和财务建模
- [ ] business-analyst 可执行业务流程分析和需求获取
- [ ] 所有输出文档格式结构化、可测试、无歧义
- [ ] 与 design-kit 和 dev-kit 交接协议完整定义
